# API Rules — Role-Based Access Control for Orbita

## Summary

Orbita is a **multi-tenant project management application** built on PocketBase. Currently, every collection has wide-open API rules (`""` on all operations), which is a critical security risk. This RFC defines a **privacy-first, role-based permission model** implemented entirely through PocketBase API rules and server-side hooks — no application-level middleware required.

## Permission Philosophy

### 1. "Nobody Sees Anything By Default"

A freshly registered user sees **nothing**. They must be explicitly invited to organizations and projects. This is the opposite of "open by default".

### 2. The Org Is a Container, Not a Permission

Membership in an organization grants the **ability to be added to projects**, not automatic visibility into them:

```
Organization "Acme Inc"
├── Member: Alice (can be added to projects)
├── Member: Bob (can be added to projects)
├── Project: "Website Redesign" → members: Alice, Bob
└── Project: "Secret Initiative" → members: Alice only (Bob cannot see it)
```

### 3. Owners See Everything, Always

Organization owners retain full access to every project in their org, regardless of project membership. This is the "break glass" permission for leadership.

### 4. Roles Are Additive, Not Exclusive

A user can hold multiple roles simultaneously (e.g., org member + project admin). The most permissive role wins.

## The Five Roles

| Role | Scope | Capabilities |
|------|-------|-------------|
| **Organization Owner** | Org-wide | Full control: sees everything, manages org members, overrides any project |
| **Organization Member** | Org-wide | Can be added to projects; sees org exists but not its projects (until invited) |
| **Project Admin** | Per-project | Full project control: manages members, deletes resources, changes settings |
| **Project Member** | Per-project | Creates & edits boards, lists, cards, documents, comments. Cannot delete or manage members |
| **Project Viewer** | Per-project | Read-only: sees everything in a project, cannot change anything |

## Key Design Decisions

| # | Decision | Rationale |
|---|----------|-----------|
| 1 | **Strict project visibility**: Org members can only see projects they are explicitly added to. Org membership alone does not grant access to any project. | Privacy-first: org membership ≠ project access. See [Project Visibility Options](#project-visibility-options) below. |
| 2 | **Org owner override**: Org owners have full access to every project in their org. | Leadership needs visibility |
| 3 | **Project creation**: Any org member can create a project; creator auto-becomes project admin (via hook). | Low-friction project creation |
| 4 | **Viewer role**: Added to `project_members.role`. Viewers see everything but cannot create/update. Enforced with `?!= "viewer"` in rules. | Read-only stakeholders |
| 5 | **`is_external` flag on members**: A boolean field on `organization_members`, `project_members`, and `invitations`. Marks a person as external to the organization (e.g., a freelancer, contractor, or client). Has zero effect on permissions — the flag is purely for the UI to show a badge or filter external members. | Keeps the permission model simple. Instead of a separate "external collaborator" role or a parallel membership system, freelancers join the org as regular members with a flag. The security boundary is project-level — being flagged `is_external` doesn't grant or revoke any access. |
| 6 | **Events immutable**: `card_events` and `document_events` are append-only audit logs created by server hooks. `create/update/deleteRule: null`. | Accountability |
| 7 | **Comments**: Only the author can edit their comment. Author, project admins, and org owners can delete. | Message integrity + cleanup ability |
| 8 | **Labels**: Any non-viewer project member can create. Only project admins and org owners can update or delete (they affect all cards). | Prevents destructive label changes |
| 9 | **Self-leave**: Any member can remove themselves from an org or project. | No "trapped" accounts |
| 10 | **Last-owner protection**: Cannot remove or demote the last org owner (enforced via hook). | Prevents orphaned orgs |
| 11 | **Cascade on org leave**: When removed from an org, all project memberships in that org are removed (via hook). | Clean data |
| 12 | **Deep traversal**: Rules use up to 4 levels of relation expansion (`card.board.project.organization`), within PocketBase's 6-level limit. | No denormalization needed (for now) |
| 13 | **Public sharing deferred**: Public board and document sharing via shareable links requires a custom PocketBase route (anonymous access) and a `shares` collection. This is designed separately from the API rules in this document. | Separate concerns — API rules cannot express anonymous access patterns. |

### Project Visibility Options

There are two philosophies for how projects relate to organization membership. This design uses **Option B**.

**Option A — Org membership grants project visibility.** Every org member automatically sees every project in the org. Joining the org is the only gate; projects have no independent access control. This is simpler to manage (add someone to the org, they see everything) but makes it impossible to have confidential projects within an org — a problem for companies with HR, finance, or client-sensitive work.

**Option B — Explicit project membership required.** Org membership is a prerequisite for being added to projects, but does not by itself grant access to any project. Each project maintains its own member list. This is the model used in this design. It trades some admin overhead (members must be added to each project individually) for strong privacy boundaries within an org.

**Why Option B?** It is safer to start restrictive and loosen later. If project creation defaults to "add all org members" in the UI, the ergonomics of Option A can be achieved without compromising the security model. Moving from Option A to Option B, by contrast, would require retroactively locking down existing projects — a risky migration.

## API Rules

> **Format**: Each block is a valid PocketBase filter expression. Copy into `pb_schema.json` or the Admin UI.
>
> **IMPORTANT**: In PocketBase, `""` (empty string) means "open to everyone", while `null` means "superuser-only". Use `null` for rules that should be restricted to superusers (e.g., the event collections).

### `users` (`_pb_users_auth_`)

```json
{
  "listRule": "@request.auth.id != \"\"",
  "viewRule": "@request.auth.id != \"\"",
  "createRule": "",
  "updateRule": "@request.auth.id = id",
  "deleteRule": "@request.auth.id = id"
}
```

> **Design note**: The `listRule` allows any authenticated user to enumerate all users. This is a deliberate trade-off — it enables user search when sending invitations, but means a logged-in user can discover other users on the platform. If stricter privacy is desired, this can be narrowed to only users who share at least one organization with the requester, at the cost of a slower, more complex query.

## Required Server-Side Hooks

These hooks enforce invariants that cannot be expressed in PocketBase filter syntax alone. Install in `pb_hooks/` or the PocketBase hooks panel.

### Hook 1: Auto-add org owner on org creation

```javascript
onAfterCreateRequest((e) => {
  const org = e.record;
  const user = e.auth;

  // Skip if org is created without auth (e.g., migrations)
  if (!user) return;

  const membership = new Record();
  membership.set("user", user.id);
  membership.set("organization", org.id);
  membership.set("role", "owner");
  membership.set("is_external", false);
  $app.save(membership);
}, "organizations");
```

### Hook 2: Auto-add project admin on project creation

```javascript
onAfterCreateRequest((e) => {
  const project = e.record;
  const user = e.auth;

  if (!user) return;

  // Propagate is_external from org membership
  const orgMemberships = $app.findRecordsByFilter(
    "organization_members",
    `user="${user.id}" && organization="${project.get("organization")}"`,
    "",
    1
  );

  const isExternal = orgMemberships.length > 0
    ? orgMemberships[0].get("is_external")
    : false;

  const membership = new Record();
  membership.set("user", user.id);
  membership.set("project", project.id);
  membership.set("role", "admin");
  membership.set("is_external", isExternal);
  $app.save(membership);
}, "projects");
```

### Hook 3: Last-owner protection

Prevents removal or demotion of the last organization owner. The delete path checks the count of remaining owners before allowing removal. The update path fetches the original record from the database to compare the old role against the new role — a necessary step because `e.record` in an update hook contains the incoming data, not the stored values.

```javascript
// Prevent deleting the last owner
onBeforeDeleteRequest((e) => {
  const member = e.record;
  if (member.get("role") !== "owner") return;

  const owners = $app.findRecordsByFilter(
    "organization_members",
    `organization="${member.get("organization")}" && role="owner"`,
    "",
    0
  );

  if (owners.length <= 1) {
    throw new BadRequestError("Cannot remove the last organization owner");
  }
}, "organization_members");

// Prevent demoting the last owner
onBeforeUpdateRequest((e) => {
  const member = e.record;
  const newRole = member.get("role");

  // Fetch the original record to compare old vs new role
  const original = $app.findRecordById("organization_members", member.id);
  const oldRole = original.get("role");

  // If role is being changed away from "owner"
  if (oldRole === "owner" && newRole !== "owner") {
    const owners = $app.findRecordsByFilter(
      "organization_members",
      `organization="${member.get("organization")}" && role="owner"`,
      "",
      0
    );

    if (owners.length <= 1) {
      throw new BadRequestError("Cannot demote the last organization owner");
    }
  }
}, "organization_members");
```

### Hook 4: Cascade — remove project memberships when leaving org

```javascript
onAfterDeleteRequest((e) => {
  const member = e.record;
  const userId = member.get("user");
  const orgId = member.get("organization");

  const projects = $app.findRecordsByFilter(
    "projects",
    `organization="${orgId}"`,
    "",
    0
  );

  const projectIds = projects.map((p) => p.id);

  for (const projectId of projectIds) {
    const projectMemberships = $app.findRecordsByFilter(
      "project_members",
      `user="${userId}" && project="${projectId}"`,
      "",
      0
    );

    for (const pm of projectMemberships) {
      $app.delete(pm);
    }
  }
}, "organization_members");
```
