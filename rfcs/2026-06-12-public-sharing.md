# Public Sharing — Read-Only Board & Document Access

## Summary

Allow project members to generate **public, read-only links** to boards and documents. These links work without authentication — anyone with the URL can view the shared resource. Unlike the standard API rules (which require an authenticated user), public sharing requires a fundamentally different approach: a **custom PocketBase route** that validates a share token and returns data directly, bypassing the normal permission system entirely.

## Why a Custom Route?

The standard PocketBase API rules work on a simple principle: every request is authenticated, and rules filter what the authenticated user can see. Public sharing inverts that — there is **no authenticated user**. PocketBase does not support per-record anonymous access in its filter syntax (there is no `@request.shareToken` or similar), so we handle public access through a custom server-side route that validates the share token and returns data directly.

The `shares` collection itself is **only for management** (create, view, revoke shares). Its API rules use the standard auth-based permission model.

## The `shares` Collection

| Field | Type | Details |
|-------|------|---------|
| `token` | text | Autogenerated 64-char hex string (Hook 1) |
| `resource_type` | select | `["board","document"]` |
| `resource_id` | text | The ID of the shared board or document |
| `password` | password | Optional — if set, viewer must enter password |
| `expires_at` | date | Optional — share link stops working after this date |
| `created_by` | relation → users | Who created the share |
| `is_active` | bool | Default `true`. Set to `false` to revoke. |
| `allow_comments` | bool | Default `false`. Reserved for future use. |

### API Rules (Management Only)

These rules govern who can **create, view, update, or delete** share records. They do **not** control who can access the shared content — that is handled by the custom route.

```json
{
  "listRule": null,
  "viewRule": "@request.auth.id != \"\" && created_by.id = @request.auth.id",
  "createRule": null,
  "updateRule": "@request.auth.id != \"\" && created_by.id = @request.auth.id",
  "deleteRule": "@request.auth.id != \"\" && created_by.id = @request.auth.id"
}
```

> **Design note**: `listRule: null` (superuser-only) and `createRule: null` (superuser-only) mean share management happens through the custom route or application logic, not direct API calls. This prevents enumeration of all shares. If you want self-service share management via the API, see the "Alternatives" section below.

### Hook 1: Auto-generate share token

```javascript
onBeforeCreateRequest((e) => {
  const token = [...crypto.getRandomValues(new Uint8Array(32))]
    .map(b => b.toString(16).padStart(2, '0'))
    .join('');
  e.record.set("token", token);
}, "shares");
```

## Custom Route Design

### Route: `GET /api/public/share/:token`

This is a PocketBase custom route registered in `pb_hooks/`.

**Behavior:**

1. Look up the `shares` record by `token`
2. Validate:
   - Share exists and `is_active = true`
   - Not expired (`expires_at` is null or in the future)
   - Password matches (if `password` is set, check the `password` query param or header)
3. Based on `resource_type`, fetch the resource and return it as JSON
4. For `board`: return the board + its lists + its cards (the full board view)
5. For `document`: return the document with its content

**Pseudocode:**

```javascript
router.get("/public/share/:token", (c) => {
  const token = c.req.param("token");
  
  // 1. Find the share record
  const share = $app.findFirstRecordByData("shares", "token", token);
  if (!share || !share.get("is_active")) {
    throw new NotFoundError("Share link not found or inactive");
  }
  
  // 2. Check expiration
  const expiresAt = share.get("expires_at");
  if (expiresAt && new Date(expiresAt) < new Date()) {
    throw new BadRequestError("Share link has expired");
  }
  
  // 3. Check password (if set)
  const password = share.get("password");
  if (password) {
    const providedPassword = c.req.query("password") || c.req.header("X-Share-Password");
    // Compare using PocketBase's password verification
    if (!providedPassword || !$app.passwordValidate(password, providedPassword)) {
      throw new UnauthorizedError("Invalid or missing password");
    }
  }
  
  // 4. Fetch the resource
  const resourceType = share.get("resource_type");
  const resourceId = share.get("resource_id");
  
  if (resourceType === "board") {
    const board = $app.findRecordById("boards", resourceId);
    const lists = $app.findRecordsByFilter("lists", `board="${resourceId}"`);
    // Fetch cards for each list...
    // Return combined data
    return c.json({ board, lists, cards: /* ... */ });
  }
  
  if (resourceType === "document") {
    const doc = $app.findRecordById("documents", resourceId);
    return c.json({ document: doc });
  }
  
  throw new BadRequestError("Unknown resource type");
});
```

## Frontend Integration

The frontend needs:

1. **A "Share" button** on boards and documents that creates a `shares` record (via the custom route or app logic) and displays the public URL
2. **A share management panel** — list active shares, revoke, set password/expiration
3. **A public view route** — `/:token` that renders the shared content without requiring authentication

## Security Considerations

| Concern | Mitigation |
|---------|-----------|
| Token enumeration | 64-char hex = 256 bits of entropy. Brute-force infeasible. |
| Token leaking in logs | Never log the full token; log only first 8 chars for debugging. |
| Password brute-force | PocketBase's password field uses bcrypt. Rate-limit the custom route. |
| Expired shares still accessible | Checked on every request. |
| Shared content changes after sharing | This is **by design** — the link shows the live resource, not a snapshot. |
| Cascading access via board | When sharing a board, all lists, cards, and labels on that board become visible. Document this clearly in the UI. |

## Alternatives & Future Enhancements

### Self-service share management via API

If you want users to manage their own shares through the standard PocketBase API (rather than a custom route for everything), change the `shares` rules to:

```json
{
  "listRule": "@request.auth.id != \"\" && created_by.id = @request.auth.id",
  "viewRule": "@request.auth.id != \"\" && created_by.id = @request.auth.id",
  "createRule": "@request.auth.id != \"\" && (
    @collection.organization_members.user ?= @request.auth.id
    && @collection.organization_members.organization ?= /* ??? */
    && @collection.organization_members.role ?= \"owner\"
    || @collection.project_members.user ?= @request.auth.id
    && @collection.project_members.project ?= /* ??? */
    && @collection.project_members.role ?!= \"viewer\"
  )",
  "updateRule": "@request.auth.id != \"\" && created_by.id = @request.auth.id",
  "deleteRule": "@request.auth.id != \"\" && created_by.id = @request.auth.id"
}
```

**The problem**: `shares` has `resource_type` (a string) and `resource_id` (a string), not relation fields. You cannot traverse from `shares` → `boards` → `project` → `organization` in PocketBase filter syntax because there is no relation field to follow. 

**Solutions:**

1. **Keep `createRule: null`** and create shares through a custom route that validates permissions in application code (recommended for now)
2. **Denormalize** — add `project` and `organization` relation fields to `shares`, set them via hook on create, and use those in the createRule
3. **Separate collections** — `board_shares` and `document_shares` each with a direct relation to their resource

### Future: Allow comments on shared boards

The `allow_comments` field is reserved for this. It would require:
- A separate custom route for submitting comments (e.g., `POST /api/public/share/:token/comments`)
- Storing a `guest_name` on the comment instead of a user relation
- Moderation features (flagging, requiring approval)

### Future: Passwordless share links

Already supported by making `password` optional. If not set, the link works without a password.
